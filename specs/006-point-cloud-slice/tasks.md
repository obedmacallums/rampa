# Tasks: Point Cloud Slice

**Input**: Design documents from `/specs/006-point-cloud-slice/`

- **[P]**: Can run in parallel (different files, no dependency on incomplete work)

**Goal**: A user traces a polyline over the cloud in a top-down orthographic view, sets a band width, and reads a chart of distance against elevation — with live width and exaggeration controls, colouring, keyboard navigation, and an honest point-count/LOD readout.

**Independent Test**: Load a survey with a point cloud, trace a line across a known feature, and confirm the chart shows terrain geometry consistent with that feature — including detail the DEM flattens.

## Dependencies & Execution Order

### Phase Dependencies

- **Setup (Phase 1)**: No dependencies. The `geotiff` install and the Potree patch are
  independent of each other and can run in parallel.
- **Foundational (Phase 2)**: Depends on Setup. **Blocks every user story.**
- **User Stories (Phases 3–6)**: All depend on Foundational.
- **Polish (Phase 7)**: Depends on the stories you intend to ship.

### A note on story independence

The template's ideal is fully independent stories. That is only partly true here, and
pretending otherwise would mislead whoever picks this up:

- **US1 is genuinely independent** and is a complete, shippable product on its own.
- **US2, US3 and US4 all extend US1's components** — `SliceChart.tsx`, `SliceControls.tsx`,
  `SlicePanel.tsx` and the store. They cannot be built before US1, and they cannot be built
  in parallel with each other without conflicting on those same files.
- `density.ts` (including `loadDecision` and `estimatePointCount`, which US4 consumes) is
  created complete in **US1**, because splitting one small pure module across two phases
  would cost more than it saves.

This is intrinsic to the feature: every story after the first adds a capability *to the
chart* that US1 creates. Sequential delivery in priority order is the honest plan.

### Within each story

- Test tasks precede their implementation task and must fail first.
- Pure modules (`domain/slice/*`) before the components that consume them.
- Components before the wiring that mounts them.

### Parallel Opportunities

- Setup: the `geotiff` install and the `potree.js` patch touch different files.
- Foundational: `types.ts`, the store, and `potreeSlice.ts` are separate files, though the
  latter two both import the types.
- US1: the three pure-module test suites (`slice-geometry`, `slice-density`, `slice-binning`)
  are independent of each other and of everything else — the largest parallel block here.
- Polish: the quickstart scenarios are independent of one another once the stack is up.

## Implementation Strategy

### MVP first (User Story 1 only)

1. Phase 1: Setup
2. Phase 2: Foundational — blocks everything
3. Phase 3: User Story 1
4. **STOP and VALIDATE** with quickstart.md Scenarios 1, 2 and 7
5. Ship it. A working slice tool is useful with no overlay, no measurement and no export.

### Incremental delivery

Setup + Foundational → US1 (MVP) → US2 (raster audit) → US3 (measurement) → US4 (export),
validating each against its quickstart scenario before starting the next.

If only one story after the MVP can be built, **US2 is the one worth having**: the DEM
overlay is what makes the tool an instrument for judging the raster that every future
analysis in this platform depends on.
